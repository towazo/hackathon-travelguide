# 役割分担 全体共有事項（はじめに必ず読むページ）

このページは、A〜D担当の4人全員が作業を始める前に、必ず目を通してください。
プログラミングが初めての人でも読み進められるように、専門用語には説明をつけています。

---

## 1. これから作るサイトはどんなものか

**「現在地」と「目的地」を入力すると、その移動の途中に立ち寄れる『寄り道スポット』をAIが提案してくれるサイト**を作ります。

具体例で説明します。

1. ユーザーが画面の入力欄に「現在地：新宿駅」「目的地：浅草」と入力する
2. ユーザーが「検索」ボタンを押す
3. サイトの裏側でAI（人工知能）に「新宿駅から浅草までの間で、寄り道できそうな場所を教えて」と問い合わせる
4. AIが「上野公園」「秋葉原」「神田周辺のカフェ」のような候補を、説明文とおすすめ理由つきで返してくる
5. その候補を画面にカードのような見た目で並べて表示する

この一連の流れを、4人で手分けして作ります。

---

## 2. 「コンポーネント」という考え方

Reactというライブラリ（プログラムの部品集）を使ってこのサイトを作ります。Reactでは、画面を1つの大きなプログラムとして書くのではなく、**小さな部品（コンポーネント）に分けて組み合わせる**というやり方をします。

レゴブロックをイメージしてください。「入力フォームのブロック」「スポット1件分のカードのブロック」のように、小さいブロックを別々に作っておいて、最後に組み合わせて1つの作品（サイト）にする、というイメージです。

今回のサイトは、以下の5つの部品（＋1つのAI連携部分）でできています。

```
App（全体をまとめる係）
├── SpotSearchForm（入力フォームの部品）……担当A
├── SpotResultList（一覧表示の部品）………担当B
│     └── SpotCard（1件分の表示の部品）…担当B
└── aiSpotService（AIに問い合わせる処理）…担当C
```

`App`は部品そのものではなく、「どの部品をどこに配置するか」「部品同士でどうデータをやり取りするか」を管理する司令塔の役割です。これを担当Dが受け持ちます。

---

## 3. データがどう流れるか（全体の流れ図）

「①ユーザーの入力」から「④画面への表示」まで、データがどう受け渡しされるかを図にしました。矢印の上に書いてあるのが、実際にやり取りされるデータです。

```
【ユーザーの操作】
    ↓ 「新宿駅」「浅草」と入力してボタンを押す
┌─────────────────────────┐
│ SpotSearchForm（担当A）        │
└─────────────────────────┘
    ↓ onSearch("新宿駅", "浅草") を呼び出す
      （＝「これが入力されたよ」と親に伝える）
┌─────────────────────────┐
│ App（担当D）                  │
│ ・loading を true にする（読込中）  │
└─────────────────────────┘
    ↓ getSpots("新宿駅", "浅草") を呼び出す
┌─────────────────────────┐
│ aiSpotService（担当C）         │
│ ・AIに問い合わせる                │
│ ・AIの返事を整理する               │
└─────────────────────────┘
    ↓ [{name, description, reason}, ...] を返す
┌─────────────────────────┐
│ App（担当D）                  │
│ ・spots にAIの返事を保存する         │
│ ・loading を false にする         │
└─────────────────────────┘
    ↓ spots を渡す
┌─────────────────────────┐
│ SpotResultList（担当B）        │
│ ・spots を1件ずつ取り出す           │
└─────────────────────────┘
    ↓ spot（1件分）を渡す
┌─────────────────────────┐
│ SpotCard（担当B）             │
│ ・名前・説明・理由を画面に表示          │
└─────────────────────────┘
```

このように、**データは基本的に上から下へ、そして最後にまた画面へ返ってくる**、という一方通行の流れになっています。この流れさえ守れば、4人がバラバラに作業しても、最後にきちんと組み合わさります。

## 4. 最低限の用語集（プログラミングが初めての人向け）

各担当の手順書の中に出てくる言葉を、先にまとめて説明します。分からなくなったら、このページに戻ってきてください。

| 用語 | 説明 |
|---|---|
| **コンポーネント** | 画面の部品のこと。上記の`SpotSearchForm`や`SpotCard`など。1つの部品は1つのファイル（`.jsx`）に対応します。 |
| **props（プロップス）** | 親のコンポーネントから子のコンポーネントに渡す「データ」や「関数」のこと。関数を呼ぶときの引数（渡す値）のようなものだとイメージしてください。 |
| **state（ステート）／useState** | コンポーネントが「覚えておく必要がある値」のこと。例えば入力欄に今何が入力されているか、AIの返事がまだ来ていないか、などです。`useState`は、その値を覚えておくための道具（React標準の関数）です。値が変わると画面が自動的に描き直されます。 |
| **JSX（ジェイエスエックス）** | JavaScriptの中にHTMLのような見た目のタグ（`<div>`や`<button>`など）を直接書ける書き方のことです。Reactのコンポーネントは主にこの書き方で「画面に何を表示するか」を書きます。 |
| **イベント（onClickやonChangeなど）** | 「ボタンが押された」「入力欄の文字が変わった」のような、ユーザーの操作のきっかけのことです。`onClick={関数}`と書くと、ボタンが押されたときにその関数が実行されます。 |
| **非同期処理（async/await）** | 「時間がかかる処理が終わるのを待ってから、次の処理を行う」という書き方です。AIへの問い合わせはすぐに終わらない（数秒かかる）ため、この書き方を使います。 |
| **JSON（ジェイソン）** | データを文字列として表現するときの、よく使われる決まった書き方（フォーマット）です。例：`{ "name": "上野公園" }`。AIとのやり取りでよく登場します。 |
| **API（エーピーアイ）** | あるプログラムが、別のプログラム（今回はAIのサービス）に対して「これをやってほしい」とお願いするための窓口のことです。 |

## 5. データの形（全員共通のお約束）

AIが提案する「寄り道スポット」は、以下の形のオブジェクト（`{ }`で囲まれた、名前と値のセットのかたまり）の配列（`[ ]`で囲まれた、複数のデータをまとめたもの）とします。

```js
[
  { name: "上野公園", description: "自然を楽しみながら休憩できる", reason: "移動ルートの近くにあり気分転換になるため" },
  { name: "秋葉原", description: "買い物や観光に立ち寄りやすい", reason: "乗り換え駅から近いため" },
]
```

それぞれの項目の意味：

- `name`：スポットの名前（文字列）。画面には大きめの見出しとして表示します。
- `description`：どんな場所かの説明（文字列）。
- `reason`：AIが「なぜここをおすすめするか」を説明する文章（文字列）。

**なぜ最初に決めるのか**：担当A・B・C・Dは、それぞれ別のファイルを別々のタイミングで作ります。もし「AIから返ってくるデータの形」がバラバラだと、最後に組み合わせたときにエラーだらけになります。先に「この形で統一する」と決めておくことで、各担当が独立して作業を進められます。

この配列を、担当Dが `spots` という名前のstateで管理し、担当Bのコンポーネントに渡します。

## 6. 関数・propsの名前の約束

コンポーネント同士やファイル同士でデータをやり取りするときに使う「名前」も、先に決めておきます。名前がバラバラだと、他の人が作った部品とうまくつながりません。

| 名前 | 使う場所（誰から誰へ） | 内容 |
|---|---|---|
| `onSearch(from, to)` | SpotSearchForm → App | フォームのボタンが押されたときに、Appに「現在地(from)と目的地(to)が入力されたよ」と伝えるための関数。担当Dが用意し、担当Aが呼び出します。 |
| `getSpots(from, to)` | aiSpotService.js | 現在地と目的地を受け取り、AIに問い合わせて`spots`の形の配列を返す関数（非同期処理）。担当Cが作ります。 |
| `spots` | App → SpotResultList | 画面に表示する、寄り道スポットの配列。 |
| `spot` | SpotResultList → SpotCard | `spots`の中の1件分のオブジェクト。 |
| `loading` | App内部で管理 | AIの返事を待っている最中かどうか（`true`／`false`）。待っている間は画面に「探しています…」のような表示を出すために使います。 |
| `error` | App内部で管理 | 何か失敗したときのエラーメッセージ（文字列）。失敗していなければ`null`（何もない状態）。 |

**この表の名前は、途中で勝手に変更しないでください。** 変更したくなった場合は、必ず4人全員に共有してから行ってください。1人が名前を変えると、他の3人の部品がつながらなくなります。

## 8. 進め方のコツ（全体スケジュール感）

1. **まず全員でこのページの内容（データの形・名前の約束）に合意する**
2. 担当Cが、本物のAIの代わりに「決まった形の仮のデータ」を返すだけの仮の関数（ダミー版）を先にさっと作る（詳細は[03_C_ai.md](./03_C_ai.md)参照）
3. そのダミー版を使って、担当A・B・Dが自分の部品を作り、画面上で動くところまで確認する
4. その間に担当Cは、本物のAIへの問い合わせ処理をじっくり作り込む
5. 最後に、ダミー版を本物のAI処理に差し替えて、全体を通しで動作確認する

**ポイント：本物のAI連携が完成するのを待たずに、他の3人が作業を進められるようにする**のがこの進め方の狙いです。ハッカソンのような時間の限られた場では、この「先にダミーで動かす」考え方がとても重要です。

完成の目安：「新宿駅」「浅草」と入力してボタンを押すと、寄り道スポットが3件くらい画面に表示される状態になれば、最低限の機能は完成です。

---

## 9. 困ったときは

- 自分のファイルの中だけで悩む前に、まず「7. 名前の約束」の表に立ち返り、propsや関数の名前が合っているか確認してください
- 「渡ってくるはずのデータが`undefined`になる」といったエラーの多くは、propsの名前のスペルミスや、渡し忘れが原因です
- それでも解決しない場合は、隣の担当（データを渡す相手・受け取る相手）に直接確認するのが一番早いです
