## 3. 新しいデータ契約（全員共通・最重要）

これまで`onSearch(from, to)`・`getSpots(from, to)`のように、現在地と目的地の2つの値だけをやり取りしていました。今回から項目が増えるため、**1つのオブジェクト（まとまったデータ）**にまとめてやり取りする形に変えます。

### 3-1. 検索条件（SpotSearchForm → App → aiSpotService → サーバー）

```js
{
  from: "新宿駅",
  to: "浅草",
  date: "2026-07-05",   // "YYYY-MM-DD"形式の文字列。何も選んでいなければ当日の日付
  meal: true,             // 食事ができる場所を希望する場合 true
  indoorOnly: false,      // 屋内の場所を希望する場合 true
  parking: false,         // 駐車場がある場所を希望する場合 true
}
```

- チェックボックスは「チェックを入れた＝その条件を希望する」という意味にします。チェックを入れなければ`false`（＝特に希望しない、AIにおまかせ）という扱いです。
- `meal`・`indoorOnly`・`parking`という名前は、この後の手順書すべてで共通して使うので、スペルを変えないでください。

### 3-2. スポット情報（AI → App → SpotResultList → SpotCard）

既存の3項目に、以下の5項目を追加します。

```js
{
  name: "上野公園",
  description: "自然を楽しみながら休憩できる",
  reason: "移動ルートの近くにあり気分転換になるため",
  nearestStation: "上野駅",         // 最寄り駅
  travelTime: "電車で約15分",         // 現在地からの移動時間の目安
  budget: "0円〜500円",               // 推定予算の目安
  stayDuration: "30分〜1時間",         // 推定滞在時間の目安
  link: "https://www.google.com/maps/search/?api=1&query=上野公園", // 詳細を調べるためのリンク
}
```

- `travelTime`・`budget`・`stayDuration`は、正確な計算ではなくAIによる**目安・推定**です（最初の設計で「正確なルート検索」は対象外としたことと矛盾しません）。
- `link`の作り方には注意点があります。次の項目で説明します。

## 4. `link`（スポットのリンク）についての重要な注意

AIに「スポットの詳細ページのURLを教えて」と直接尋ねると、**実在しないURL（もっともらしいが嘘のリンク）を作り出してしまう**ことがあります。これは「ハルシネーション」と呼ばれるAI特有の失敗パターンです。

そのため、**AIにはURLそのものを生成させず、スポット名だけを返してもらい、リンクはプログラム側で機械的に組み立てます**。具体的には、Googleマップの検索結果ページに飛ぶURLを、スポット名から自動生成します（詳しい実装は[08_C_ai_v2.md](./08_C_ai_v2.md)に書いています）。この方式なら、リンク切れや存在しないページに飛ぶ心配がありません。

## 7. 完成の目安

- 日付欄に当日の日付が最初から入っている
- 3つのチェックボックスが操作できる
